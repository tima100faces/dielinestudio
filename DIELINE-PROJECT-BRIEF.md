# Dieline Generator — бриф для старта проекта (Claude Code)

> Документ для старта. Геометрию строим **с нуля и корректно** по определению каждого
> типа коробки. Прошлые черновые наброски развёрток НЕ используем как образец —
> в них были ошибки в геометрии. Источник истины — реальные продакшн-файлы и
> стандарты FEFCO / ECMA, а не предыдущий код.

## 1. Цель
Свой генератор дайлайнов (развёрток) упаковки. Пользователь вводит размеры коробки и
толщину картона — получает готовый векторный файл для печати и высечки. Платные
сервисы (pacdora, diecuttemplates) дорогие и подписочные; нам нужен свой инструмент
под несколько типов коробок, реально используемых в работе.

## 2. Почему это реально
Стандартные коробки описываются каталогами FEFCO и ECMA — это конечный набор типов,
каждый из которых задаётся жёсткой параметрической формулой развёртки от 3–4 величин
(ширина, глубина, высота, толщина картона). Это не «рисование», а детерминированная
геометрия. Сложность не в математике, а в производственных деталях (см. §5).

## 3. Стек и архитектура
- **Ядро — Python.** Библиотека `ezdxf` для вывода DXF с настоящими `LINE` / `ARC` /
  `CIRCLE` / `LWPOLYLINE` и слоями (не полилинейная аппроксимация). При необходимости
  `shapely` для офсетов/булевых операций (прорези, компенсация толщины).
- **Архитектура «один тип = один сборщик».** Общий слой геометрии (примитивы → DXF/SVG/PDF,
  слои, цвета, рамка) переиспользуется; каждый тип коробки — отдельная функция/класс,
  возвращающая набор примитивов. Добавление типа не трогает экспорт и UI.
- **Веб-панель** (Flask или FastAPI): форма размеров → живое SVG-превью → скачивание
  DXF / PDF / SVG. Разворачивается на VPS пользователя.
- **Превью на сервере** через рендер SVG в PNG (`cairosvg`) — чтобы агент мог визуально
  проверять результат (см. §9).

## 4. Требования к выводу
- Форматы: **DXF** (основной, для высечки), **PDF** (вектор, для Illustrator), **SVG** (превью/правка).
- Единицы — **мм**, масштаб 1:1.
- **Слои:** `CUT` (рез), `CREASE` (биговка, пунктирный тип линии), `INFO` (титульный блок).
- **Цвета (конвенция продакшна):** рез — красный (ACI 1), биговка — зелёная (ACI 3).
- В DXF — **истинные дуги и окружности**, а не ломаные. 0 ошибок при `doc.audit()`.

## 5. Производственные правила (выверено на реальных файлах) — КЛЮЧЕВОЕ
Это то, что отличает рабочий дайлайн от «контура коробки». Заложить с самого начала:

1. **Толщина живёт в геометрии, а не только в габаритах.** Где картон сгибается на 180°
   (подгиб/фальц) — там **двойная биговка**: две параллельные линии сгиба с зазором
   ≈ толщине картона. Где панель оборачивает другую — её положение смещается на толщину
   (score-to-score). Просто увеличить наружный размер недостаточно.
2. **Прорези функциональные, не декоративные.** Два механизма:
   - язычки подгиба/фальца входят в **прорези в смежной панели** → стенка становится
     двойной и держится без клея;
   - замковый язычок (например каплевидный, teardrop) **вставляется в ответную прорезь**
     на корпусе и фиксирует крышку.
   Прорезь и язычок должны совпадать по положению и размеру с учётом сгибов.
3. **Не все линии прямые.** Скруглённые радиусы по углам, треугольные/косые угловые
   клапаны, угловые клиренсы — это снижает риск разрыва картона и облегчает сборку.
4. **Титульный блок** (слой `INFO`): артикул, внутренние размеры (Ш×Г×В), цвет, допуск ±,
   дата. Это обязательная мета на реальном дайлайне.
5. **Допуск и ручная резка.** В цеху допуск порядка ±5 мм, часть лишнего обрезается на
   столе вручную — абсолютная точность не критична, важнее правильная механика и читаемость.

## 6. Что обязательно в каждом типе коробки
Для конкретного типа сборщик должен корректно выдавать: внешний контур (рез), все линии
сгиба (биговка, с двойной биговкой на 180° фальцах), функциональные прорези и ответные
язычки, угловые клапаны/клиренсы, скругления, и (если предусмотрено типом) палец-вырез
для открывания. Геометрию каждого типа выводим из его определения FEFCO/ECMA и сверяем
с реальным образцом. **Ничего лишнего, чего нет в задании, не добавлять** (например,
вентиляционные отверстия — только если они явно нужны).

## 7. Первый тип и дорожная карта
- **Первым делать тот тип, для которого есть реальный эталонный файл** — его развёртку
  можно положить рядом и сверять размеры и механику 1:1 (самый надёжный способ проверки).
  У пользователя есть продакшн-дайлайн плоской коробки **430×260×30** (tuck-end, под
  LED-панель) — хороший кандидат на тип №1.
- Дальше по каталогу: mailer box (FEFCO 0427), tray box с крышкой (FEFCO 0426),
  food box (ECMA B20.01.00.53). Каждый — отдельный сборщик по общей схеме.

## 8. Структура репозитория (предложение)
```
dieline/
  core/        # примитивы, слои, экспорт DXF/SVG/PDF, рендер превью
  boxes/       # по файлу на тип: mailer.py, tray.py, foodbox.py, ...
  web/         # Flask/FastAPI: форма + превью + скачивание
  samples/     # эталонные реальные дайлайны для сверки
  tests/       # геометрические проверки + аудит DXF
  CLAUDE.md
```

## 9. Как проверять результат (обязательно на каждом шаге)
- После генерации **рендерить SVG → PNG и смотреть глазами**: контур замкнут, сгибы на
  местах, прорези совпадают с язычками, ничего не торчит и не пересекается.
- Прогонять `ezdxf.readfile(...).audit()` — должно быть **0 ошибок**; проверять, что в DXF
  именно `ARC`/`CIRCLE`, а не аппроксимация ломаными.
- Где есть эталон — сверять ключевые размеры развёртки с реальным файлом.

## 10. Частые ошибки агента (избегать) — в CLAUDE.md
- Закладывать толщину только в наружный размер, забывая двойную биговку на фальцах.
- Делать прорези декоративными, без совпадающего ответного язычка.
- Аппроксимировать дуги ломаными в DXF вместо настоящих `ARC`.
- Добавлять элементы, которых нет в задании на конкретный тип.
- Путать слои `CUT` и `CREASE` или их цвета.
- Не проверять развёртку визуальным рендером перед тем, как считать шаг готовым.

## 11. Калибровка под картон
Зазор двойной биговки и глубину фальца держать **отдельными константами** в начале
каждого сборщика. После одной пробной высечки/сборки образца они подкручиваются одним
числом под конкретную марку картона.
